# 子代理、轮次限制与失败保护设计方案

日期：2026-06-12  
项目：`learn-agent`  
文档状态：设计确认稿，等待 review

## 1. 背景

当前项目是一个极简 coding agent CLI，核心流程位于 `src/learn_agent/agent_loop.py`：

```text
main.py
  → LoopState(messages=history)
  → agent_loop(state)
  → Anthropic Messages API
  → assistant 返回 text/tool_use
  → execute_tool_use_blocks 执行工具
  → tool_result 追加回 messages
  → 继续下一轮，直到没有 tool_use
```

现有实现已经具备：

- Anthropic tool use 调用链路；
- 工具注册中心：`src/learn_agent/tools/register_tools.py`；
- 文件读取、写入、编辑、glob、bash 等基础工具；
- `create_plan` / `update_plan_status` 计划系统；
- `LoopState` 维护单次 agent loop 的运行状态；
- `normalize_messages` 处理 Anthropic 消息格式约束。

但当前存在几个限制：

1. 父 agent 自己承担所有信息收集与执行动作，容易把大量文件内容、搜索结果和工具轨迹塞进同一个上下文。
2. `agent_loop(state)` 没有最大轮次限制，模型如果持续调用工具，可能无限循环。
3. 工具失败后没有统一失败预算和安全退出机制。
4. 当前 `LoopState` 的运行态生命周期没有显式方法表达，后续新增失败统计、停止原因后需要更清晰的重置边界。

本方案目标是在不引入复杂多 agent runtime 的前提下，为当前项目增加一个 MVP 级子代理能力，并同时补上最大轮次限制和失败保护。

## 2. 目标

本次设计要实现的目标：

1. 父 agent 可以把一个具体的信息收集任务派发给子 agent。
2. 子 agent 与父 agent 上下文隔离，默认不复制父 agent 的完整消息历史。
3. 子 agent 只能使用受限工具，MVP 阶段只允许读取/搜集信息。
4. 子 agent 不能再派生子 agent，避免递归和失控。
5. 父 agent 和子 agent 都有最大轮次限制。
6. 父 agent 和子 agent 都有失败预算，工具失败后允许模型换方式尝试，但超过预算后安全退出。
7. 明确 `LoopState` 中运行态字段的重置方式，避免一次请求的状态污染下一次请求。

## 3. 非目标

本次设计明确不做：

1. 不做多个子 agent 并行执行，只预留未来扩展空间。
2. 不做跨用户请求的全局 session 状态。
3. 不做长期失败记忆。
4. 不做精确到 `strategy` / `attempt` 维度的失败统计。
5. 不让子 agent 写文件、编辑文件、执行 bash、创建计划或更新计划。
6. 不自动 fork 父 agent 的完整 conversation history。
7. 不做 history 压缩或摘要化；只在代码中标注后续需要改进的原因。

## 4. 推荐方案概览

采用“受限版 agent loop 子代理”方案。

父 agent 新增一个工具：`delegate_task`。当父 agent 判断某个任务是可隔离的信息收集任务时，调用该工具。`delegate_task` 内部创建一个新的子 `LoopState`，使用子 agent 专属配置运行一段受限 agent loop。子 agent 完成后，将摘要结果作为 `tool_result` 返回父 agent。

整体结构：

```text
父 agent LoopState
  ↓
父 agent 调用 delegate_task 工具
  ↓
delegate_task 创建子 agent LoopState
  ↓
子 agent 使用受限工具 glob/read_file 自己循环
  ↓
子 agent 返回信息摘要或失败摘要
  ↓
父 agent 收到 tool_result，继续决定是否修改文件、执行命令或回复用户
```

核心原则：

- 子 agent 默认空白上下文。
- 父 agent 通过工具入参显式传递必要信息。
- 子 agent 只能使用 `glob` 和 `read_file`。
- 子 agent 不能调用 `delegate_task`。
- 子 agent 输出应是证据型摘要，而不是把所有原始文件内容完整返回给父 agent。
- 父 agent 仍然负责最终写入、编辑、命令执行和用户可见结论。

## 5. 可选方案与取舍

### 5.1 方案 A：普通工具函数，不调用 LLM

父 agent 调用 `delegate_task` 后，本地执行固定的搜索/读取逻辑。

优点：

- 最安全；
- 实现最简单；
- 没有递归风险；
- 行为确定性更强。

缺点：

- 它不是真正的 agent；
- 不会自主规划信息收集路径；
- 对复杂调查任务能力不足。

结论：不推荐作为本次主方案。

### 5.2 方案 B：受限版 agent loop

父 agent 调用 `delegate_task`，工具内部启动一个新的受限 agent loop。

优点：

- 符合“子代理”语义；
- 上下文隔离效果好；
- 能复用现有 Anthropic tool use 和 agent loop 机制；
- 子 agent 能自主使用 `glob/read_file` 多步收集信息；
- 后续可以扩展为并行子 agent。

缺点：

- 需要把当前 `agent_loop.py` 从依赖全局工具表，改造成接收 agent 配置；
- 需要补充权限过滤、轮次限制和失败预算测试。

结论：推荐作为 MVP 方案。

### 5.3 方案 C：完整多 agent runtime

引入 task queue、agent registry、并发池、任务 ID、结果聚合和生命周期管理。

优点：

- 最接近未来多个子 agent 并行运行的目标；
- 可扩展性最好。

缺点：

- 对当前项目过重；
- 会显著增加学习项目复杂度；
- 很多能力在 MVP 阶段用不上。

结论：作为后续演进方向，不作为本次实现目标。

## 6. 数据结构设计

### 6.1 `AgentConfig`

新增一个轻量配置结构，用来描述不同 agent 的能力边界和运行预算。建议放在新文件：

```text
src/learn_agent/agent_config.py
```

建议结构：

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class AgentConfig:
    name: str
    role: str  # "parent" | "subagent"
    max_turns: int
    max_failures: int
    allowed_tool_names: frozenset[str]
    can_delegate: bool = False
    system_prompt: str = ""  # role-specific system prompt
```

> `frozenset` 防止 `frozen=True` dataclass 下对 `allowed_tool_names` 的原地修改。
> `system_prompt` 由 `build_system` 根据 role 选择后注入，避免 `build_system` 内部硬编码 if-else。
> **明确：角色信息由 `AgentConfig.role` 单一来源决定**。`LoopState` 不再存储 `agent_role`，避免两处维护产生不一致。`build_system`、`execute_tool_use_blocks` 等函数统一通过 `config` 参数判断角色。

父 agent 配置：

```python
PARENT_AGENT_CONFIG = AgentConfig(
    name="parent",
    role="parent",
    max_turns=settings.PARENT_MAX_TURNS,
    max_failures=settings.PARENT_MAX_FAILURES,
    allowed_tool_names={
        "bash",
        "read_file",
        "write_file",
        "edit_file",
        "glob",
        "create_plan",
        "update_plan_status",
        "delegate_task",
    },
    can_delegate=True,
)
```

子 agent 配置：

```python
SUBAGENT_CONFIG = AgentConfig(
    name="subagent",
    role="subagent",
    max_turns=settings.SUBAGENT_MAX_TURNS,
    max_failures=settings.SUBAGENT_MAX_FAILURES,
    allowed_tool_names={"glob", "read_file"},
    can_delegate=False,
)
```

### 6.2 配置项

在 `src/learn_agent/config/settings.py` 中增加默认值：

```python
PARENT_MAX_TURNS: int = 20
PARENT_MAX_FAILURES: int = 5
SUBAGENT_MAX_TURNS: int = 6
SUBAGENT_MAX_FAILURES: int = 3
```

### 6.3 `LoopState` 扩展

当前 `LoopState` 包含 messages、turn_count、transition_reason、goal、plans。

建议扩展为：

```python
@dataclass
class LoopState:
    messages: list
    turn_count: int = 1
    transition_reason: str | None = None
    goal: str = ""
    plans: list[Plan] | None = None

    failure_count: int = 0
    consecutive_failures: int = 0
    failure_log: list[str] = field(default_factory=list)
    stopped_reason: str | None = None

    def reset_runtime_state(self) -> None:
        self.turn_count = 1
        self.transition_reason = None
        self.goal = ""
        self.plans = None
        self.failure_count = 0
        self.consecutive_failures = 0
        self.failure_log.clear()
        self.stopped_reason = None
```

字段含义：

- `failure_count`：当前 loop 内累计工具失败次数。
- `consecutive_failures`：连续失败次数，MVP 先记录，后续可用于更细熔断策略。
- `failure_log`：保存失败摘要，用于安全退出时说明原因。
- `stopped_reason`：记录安全停止原因，例如 `max_turns_exceeded` 或 `max_failures_exceeded`。

> **注意**：`agent_role` 不在此处存储，角色由 `AgentConfig.role` 单一来源确定，避免 state 与 config 信息冗余导致不一致。
>
> **`reset_runtime_state()` 的调用场景**：当前 `main.py` 每次用户输入都新建 `LoopState(messages=history)`，暂时不需要调用此方法。它预留给两个场景：
> 1. 未来如果复用同一个 `LoopState` 对象跨请求（如失败后重试同一任务，需要清空运行态但保留部分上下文）；
> 2. 子 agent 失败后父 agent 决定用新策略重试。

## 7. 状态生命周期

### 7.1 父 agent 状态

父 `LoopState` 是单次用户请求级别的运行状态。

当前 `main.py` 每次用户输入都会执行：

```python
state = LoopState(messages=history)
```

这个行为应继续保留。新增运行态字段后，每次用户请求开始时都应保证：

- `turn_count` 从 1 开始；
- `goal/plans` 为空；
- `failure_count/consecutive_failures/failure_log` 为空；
- `stopped_reason` 为空。

为了让生命周期更明确，建议在 `LoopState` 中提供 `reset_runtime_state()`。如果未来复用同一个 `LoopState` 对象，可以调用该方法重置运行态。

### 7.2 子 agent 状态

子 `LoopState` 是一次 `delegate_task` 调用级别的运行状态。

特点：

- 在 `run_delegate_task` 内部创建；
- 默认不继承父 agent 的 messages；
- 只接收父 agent 显式传入的任务包；
- 子 agent loop 完成后丢弃；
- 只把最终摘要或失败摘要返回给父 agent。

### 7.3 对话历史

`history/messages` 暂时保持现状，继续跨用户输入保留。

需要在 `main.py` 或状态创建处增加注释，说明后续改进原因：

```python
# TODO: Conversation history currently keeps full tool traces across user turns.
# This preserves context but can introduce noise and token growth.
# Consider adding history compaction/summarization after each request.
```

该注释不是本次实现范围内的待完成工作，而是记录一个明确的后续演进方向：当前保留完整 history 有利于对话连续性，但旧工具结果和旧文件内容可能逐步变成上下文噪音。

## 8. `delegate_task` 工具设计

新增工具文件：

```text
src/learn_agent/tools/run_delegate_task.py
```

工具职责：

- 接收父 agent 派发的信息收集任务；
- 构造子 agent 的初始消息；
- 使用 `SUBAGENT_CONFIG` 运行受限 agent loop；
- 提取子 agent 最终文本；
- 返回给父 agent。

工具 schema：

```json
{
  "name": "delegate_task",
  "description": "Delegate a read-only information gathering task to a restricted subagent.",
  "input_schema": {
    "type": "object",
    "properties": {
      "task": {
        "type": "string",
        "description": "The specific information-gathering task for the subagent."
      },
      "context": {
        "type": "string",
        "description": "Minimal relevant context summarized by the parent agent. Do not pass full conversation history."
      },
      "relevant_paths": {
        "type": "array",
        "items": {"type": "string"},
        "description": "Optional paths or glob hints the subagent should inspect first."
      },
      "output_format": {
        "type": "string",
        "description": "Expected output format, e.g. bullet summary, file list, findings with evidence."
      }
    },
    "required": ["task"]
  }
}
```

建议函数签名：

```python
def run_delegate_task(
    state: LoopState,
    task: str,
    context: str = "",
    relevant_paths: list[str] | None = None,
    output_format: str = "",
) -> str:
    ...
```

运行流程：

```text
run_delegate_task(parent_state, task, context, relevant_paths, output_format)
  1. 校验 task 非空。
  2. 构造子 agent 初始 user message。
  3. 创建 child_state = LoopState(messages=[user_message])。
  4. 调用 agent_loop(child_state, config=SUBAGENT_CONFIG)。
  5. 从 child_state.messages[-1].content 提取文本。
  6. 如果 child_state.stopped_reason 不为空，返回失败摘要。
  7. 否则返回子 agent findings。
```

子 agent 初始消息示例：

```text
You are a read-only research subagent.

Task:
<task>

Relevant context from parent:
<context or "(none)">

Suggested paths:
<one path per line or "(none)">

Expected output format:
<output_format or default evidence-based summary>
```

## 9. Agent loop 改造

### 9.1 函数签名

当前：

```python
def run_one_turn(state: LoopState) -> bool:
    ...

def agent_loop(state: LoopState) -> None:
    while run_one_turn(state):
        pass
```

建议改为：

```python
def run_one_turn(state: LoopState, config: AgentConfig = PARENT_AGENT_CONFIG) -> bool:
    ...

def agent_loop(state: LoopState, config: AgentConfig = PARENT_AGENT_CONFIG) -> None:
    ...
```

保留默认父 agent 配置，降低对现有测试和调用点的冲击。

### 9.2 工具过滤

新增工具过滤函数，放在 `register_tools.py` 中（与 `TOOLS` 同文件，避免跨文件隐式依赖）：

```python
# register_tools.py
def filter_tools(allowed: frozenset[str]) -> list[dict]:
    return [t for t in TOOLS if t["name"] in allowed]
```

调用 Anthropic API 时传入过滤后的工具：

```python
tools=filter_tools(config.allowed_tool_names)
```

### 9.3 工具执行权限校验

`execute_tool_use_blocks` 需要接收 config：

```python
def execute_tool_use_blocks(
    tool_use_blocks: list[dict],
    state: LoopState,
    config: AgentConfig = PARENT_AGENT_CONFIG,
) -> list[dict]:
    ...
```

执行前校验：

```python
if name not in config.allowed_tool_names:
    output = f"Error: tool '{name}' is not allowed for {config.role}"
```

这样即使模型幻觉调用了 `write_file` 或 `delegate_task`，子 agent 也无法执行。

## 10. System prompt 设计

### 10.1 父 agent prompt

父 agent 保留当前 `BASE_SYSTEM` 的核心要求，并增加子 agent 相关说明：

```text
For focused read-only investigation tasks, use delegate_task.
Delegate only tasks that can be isolated and do not require writing files.
When delegating, summarize all necessary context explicitly in the task input.
Do not assume the subagent can see the parent conversation history.
You remain responsible for final edits, command execution, and user-facing conclusions.
If a tool fails, avoid repeating the exact same failed call; try another approach.
```

### 10.2 子 agent prompt

子 agent 使用独立 prompt：

```text
You are a read-only research subagent.
You can only inspect the workspace using allowed tools.
Do not modify files.
Do not call other agents.
Do not attempt write/edit/bash operations.
Return a concise evidence-based summary to the parent agent.
Include file paths and key findings.
If a tool fails, try a different read-only approach.
Stop when the answer is sufficient.
```

`build_system(state, config)` 根据 `config` 的 `system_prompt` 注入角色 prompt，并只对 `role == "parent"` 注入 plan progress。子 agent MVP 不使用计划系统。

> **提示：子 agent 的指令分两层传达**：
> 1. `system_prompt`（L487-499）定义角色和能力边界——"You are a read-only research subagent"、禁止写文件等；
> 2. User message（L388-404）传递具体任务——task、context、relevant_paths、output_format。
>
> 这不是信息重复，而是职责分开：system prompt 管"你是谁、能做什么"，user message 管"这次要做什么"。

## 11. 最大轮次限制

当前 `agent_loop` 没有退出保护。建议改为：

```python
def agent_loop(state: LoopState, config: AgentConfig = PARENT_AGENT_CONFIG) -> None:
    while state.turn_count <= config.max_turns:
        should_continue = run_one_turn(state, config)
        if not should_continue:
            return

    state.stopped_reason = "max_turns_exceeded"
    append_safety_stop_message(state, "max_turns_exceeded")
```

默认值：

- 父 agent：20 轮；
- 子 agent：6 轮。

说明：

- 子 agent 轮次更小，避免单个信息收集任务消耗过多上下文和费用。
- 父 agent 轮次较大，允许执行复杂任务，但仍有上限。

## 12. 失败预算设计

采用“任务级失败预算 + 模型换路尝试”的策略。

### 12.1 失败识别

MVP 规则：任意 tool result 的 `content` 以 `Error:` 开头，即视为一次工具失败。

```python
def is_tool_error(content: str) -> bool:
    return content.strip().startswith("Error:")
```

### 12.2 失败统计

每轮工具执行后统计失败：

```python
failed_results = [r for r in tool_results if is_tool_error(r.get("content", ""))]

if failed_results:
    state.failure_count += len(failed_results)
    state.consecutive_failures += len(failed_results)
    state.failure_log.extend(short_error(r["content"]) for r in failed_results)
else:
    state.consecutive_failures = 0
```

### 12.3 失败截断的时刻

关键：工具失败的结果**先正常追加到 messages**，再检查预算。这样模型至少能看到最后一次失败的信息，形成完整反馈闭环。

`run_one_turn` 中的执行顺序：

```text
1. 执行所有 tool_use → 得到 tool_results
2. tool_results 追加为 user message（含错误结果）
3. 统计本轮失败 → 更新 failure_count / consecutive_failures
4. 检查 failure_count >= config.max_failures？
   - 是 → stopped_reason = "max_failures_exceeded"
           → append_safety_stop_message(state, reason)
           → return False（本次 turn 结束，不再继续循环）
   - 否 → return True（允许下一轮，模型看到错误后可换方式）
```

> 注意：失败检查在 turn 末尾进行，不在 `execute_tool_use_blocks` 内部。如果检查在工具执行中间进行，工具结果可能丢失，模型看不到发生了什么。

### 12.4 超过预算后的行为

如果 `state.failure_count >= config.max_failures`：

- 父 agent：停止当前 loop，追加安全退出说明，CLI 打印给用户。
- 子 agent：停止子 loop，返回失败摘要给父 agent；父 agent 可以换策略、自己处理或向用户说明。

默认值：

- 父 agent：5 次失败；
- 子 agent：3 次失败。

### 12.5 为什么不是立即失败退出

工具失败并不一定表示任务失败。例如：

- `read_file` 读错路径，可以先 `glob` 找正确文件；
- `glob` 没匹配，可以换模式；
- 某个文件不存在，可以读取相邻目录确认结构。

因此失败后应给模型一次换方法的机会。但为了避免无限试错，需要失败预算作为硬上限。

## 13. 安全退出消息

新增统一函数：

```python
def append_safety_stop_message(state: LoopState, reason: str) -> None:
    ...
```

安全退出消息格式：

```text
Stopped safely: <reason>.

Recent failures:
- <failure 1>
- <failure 2>
```

写入方式：

```python
state.messages.append({
    "role": "assistant",
    "content": [{"type": "text", "text": message}],
})
```

这样可以复用现有 `extract_text` 逻辑。

## 14. 计划系统与子 agent 的关系

MVP 中：

- 父 agent 可以继续使用 `create_plan` / `update_plan_status`。
- 子 agent 不使用计划工具。
- `build_system` 只在父 agent prompt 中注入 `Goal` 和 `Plan Progress`。
- 子 agent 的任务边界由父 agent 的 `delegate_task` 入参描述。

这样可以避免子 agent 自己维护计划，降低复杂度和递归风险。

## 16. 测试计划

### 16.1 配置与工具过滤

- 父 agent 工具列表包含 `delegate_task`。
- 子 agent 工具列表只包含 `glob` 和 `read_file`。
- 未授权工具不会传给 Anthropic API。

### 16.2 子 agent 权限

- 子 agent 调用 `write_file` 返回 `Error: tool ... is not allowed`。
- 子 agent 调用 `delegate_task` 返回 `Error: tool ... is not allowed`。

### 16.3 `delegate_task`

- task 为空时返回错误。
- 子 agent 初始消息包含 task、context、relevant_paths、output_format。
- 子 agent 正常返回 summary 时，父 agent 收到 summary。
- 子 agent 因 `max_turns` 或 `max_failures` 停止时，父 agent 收到失败摘要。

### 16.4 最大轮次

- 模型持续返回 tool_use 时，达到 `max_turns` 后停止。
- 停止时 `state.stopped_reason == "max_turns_exceeded"`。
- `state.messages` 最后一条为安全退出文本。

### 16.5 失败预算

- 工具返回 `Error:` 时增加 `failure_count`。
- 未超过预算时，错误照常返回给模型，允许下一轮换方式。
- 达到预算时安全退出。
- 子 agent 达到失败预算后不会影响父 agent 进程，只返回失败摘要。

### 16.6 状态重置

- `reset_runtime_state()` 清理运行态字段。
- `reset_runtime_state()` 不清理 `messages`。
- `plan_snapshot()` / `rollback_plans()` 现有行为不被破坏。

### 16.7 回归测试

- 现有测试全部通过：

## 17. 后续演进方向

本次 MVP 完成后，可以在后续阶段考虑：

## 18. 成功标准

实现完成后，应满足：

1. 父 agent 可以通过 `delegate_task` 启动子 agent 收集信息。
2. 子 agent 不继承父 agent 完整 messages，只使用父 agent 显式传入的任务包。
3. 子 agent 只能使用 `glob/read_file`。
4. 子 agent 无法调用 `delegate_task`，无法递归派生。
5. 父/子 agent 都受 `max_turns` 限制。
6. 父/子 agent 都受 `max_failures` 限制。
7. 工具失败后未超过预算时，模型可以换方式尝试。
8. 超过预算后，系统安全退出并给出原因摘要。
9. `reset_runtime_state()` 可以清理请求级运行态，不影响 history。
10. 全量测试通过。
